# #003 — Split instance status: agent reachability vs workload state

> *This was generated by AI during triage.*

**Category:** enhancement
**State:** ready-for-agent

## Reporter context

When the maintainer registered a new instance and looked at it, the API
returned `status: "unknown"`. After issuing a (silently-dropped) disable
command, it stayed `"unknown"`. There is no way to tell from the API
whether:

- The instance was registered but no agent has ever connected.
- The agent connected but no enable/disable has been issued yet.
- The agent connected, then died.
- The agent is alive and the workload is currently running.
- The agent is alive but the workload was explicitly disabled.

All five states are observably indistinguishable today, because
`InstanceStatus` is a single enum (`unknown | running | disabled`) that
conflates two orthogonal axes:

1. **Reachability** — is an agent currently online for this instance?
2. **Intended workload state** — is the operator currently asking for
   this workload to be running or stopped?

## Reproduction trace

- The instance row carries one `status` field of type `InstanceStatus`
  with values `unknown`, `running`, `disabled`.
- `unknown` is the value at registration.
- `running` is only ever set when an `enable` command's status report
  comes back successfully.
- `disabled` is only ever set when a `disable` command's status report
  comes back successfully.
- Nothing updates the field on agent connect, disconnect, crash, restart,
  or scheduled health check. There is no heartbeat path at all.

So a freshly registered instance whose agent is alive but idle reads as
`unknown`. A disabled instance whose agent has since died still reads as
`disabled`. Both are misleading.

## Agent Brief

**Category:** enhancement
**Summary:** Track agent reachability and intended workload state as two
independent fields, driven respectively by a periodic agent heartbeat and
by command-status reports.

**Current behavior:**
Single `status` field on the instance. Updated only on the completion of
a disable/enable command. Initial value `unknown`. No agent reachability
signal in the system at all.

**Desired behavior:**
The instance carries two independent pieces of state, each with its own
update rules:

**Reachability** — derived from periodic heartbeats published by the
agent on a dedicated NATS subject. Values: `unknown` (never seen), `online`
(heartbeat received within the freshness window), `offline` (no heartbeat
within the freshness window). The freshness window is short — a heartbeat
every ~10 s, considered stale after ~30 s, with the exact numbers
configurable. A timestamp of the last heartbeat is exposed on the API so
operators can sanity-check.

**Workload state** — derived from the outcome of disable/enable commands.
Values: `unknown` (never commanded), `enabled` (last successful command
was an enable, or never commanded but agent reported it running on
startup), `disabled` (last successful command was a disable). This field
does **not** flip to `unknown` if the agent goes offline; it remains the
operator's last expressed intent until a new command overrides it.

The agent publishes a heartbeat on `heartbeat.<instance_id>` immediately
after subscribing to commands, then on a fixed interval. The heartbeat
payload is small and structured (instance id, agent version, optional
metadata blob — exact shape decided by the implementer). The control
plane subscribes to `heartbeat.>`, updates the `last_heartbeat_at`
timestamp on the matching instance row, and a periodic sweeper (or a
read-side computation) marks instances stale.

**Key interfaces:**

- The Instance domain type — gains a separate workload-state field and a
  reachability indicator (the latter may be derived from
  `last_heartbeat_at` rather than stored explicitly — implementer's
  call). The existing single `status` field is split into these two,
  with care taken in the API response to keep the shape understandable
  for clients.
- The shared protocol module — gains a `HeartbeatMessage` schema and the
  subject convention `heartbeat.<instance_id>`.
- The agent — gains a background task that publishes the heartbeat on
  the documented cadence, starting right after the command subscription
  is set up.
- The control plane — gains a subscriber for `heartbeat.>` that updates
  `last_heartbeat_at` on the matching row.
- The API response for an instance — exposes both fields with stable
  names. Backwards-compatibility for the existing `status` field is not
  required; we have no clients yet.

**Acceptance criteria:**

- [ ] An instance with a live agent shows reachability `online` within
      one heartbeat cycle of the agent starting.
- [ ] Stopping the agent flips reachability to `offline` within the
      stale window.
- [ ] Issuing a disable while the agent is online sets workload state to
      `disabled` and leaves reachability untouched.
- [ ] Killing the agent after a successful disable leaves workload state
      at `disabled` and flips reachability to `offline`. Restarting the
      agent flips reachability back to `online` without changing the
      workload state.
- [ ] The API response for an instance includes the two fields and a
      `last_heartbeat_at` timestamp (or equivalent freshness indicator).
- [ ] Integration test: full lifecycle as described in the four bullets
      above, end to end.
- [ ] Heartbeat cadence and stale window are configurable, with sensible
      defaults.

**Out of scope:**

- Backfilling reachability for already-registered instances that have no
  agent — they show `unknown` until a heartbeat arrives.
- Surfacing per-project rollups (X of Y workers online) — a UI concern
  that lands separately.
- Authenticating heartbeats — same auth story as the rest of the agent
  channel; handled when auth is designed.
- Alerting on offline instances — observability is downstream; this
  issue only puts the data in place.

## Triage notes

- Independent of #001 and #002 in terms of code, but conceptually the
  third leg of the same diagnosis: once an agent self-registers (#001)
  and silently-dropped commands fail loudly (#002), heartbeats are the
  remaining gap between "what the operator sees" and "what is actually
  happening in the fleet". With all three landed, the original confusing
  observation ("status stayed unknown") becomes impossible.
- The heartbeat subject (`heartbeat.<id>`) and protocol leave room for a
  future health payload (CPU, mem, custom project metrics) without
  breaking the channel — but that expansion is its own issue.
