## Reporter context

After running the local dev stack, the maintainer registered a new instance
via `POST /api/instances` and immediately tried to disable it. Nothing
happened â€” `status` stayed `unknown`. Meanwhile a separately-running
`axis-agent` (pinned to a *different* UUID via `AXIS_AGENT_INSTANCE_ID`)
worked perfectly for commands sent to *its* UUID.

This is the UX consequence of a tighter underlying mistake: the agent and
the instance row in the control-plane DB are linked by a UUID that the
human is expected to copy by hand. The intended dev workflow is:

1. `curl POST /api/instances` to register.
2. Read the returned `id`.
3. Paste into `AXIS_AGENT_INSTANCE_ID` in `.env`.
4. Restart `axis-agent`.

Every step in (2)â€“(4) is a place to silently desync. This issue removes the
need for any of them.

## Agent Brief

**Category:** enhancement
**Summary:** Agent owns its registration. It reads identity inputs from
config, registers itself with the control plane on first start, persists
the assigned instance id locally, and on subsequent starts re-uses that id.

**Current behavior:**
`axis-agent` requires `AXIS_AGENT_INSTANCE_ID` (a UUID) to be set in the
environment. If unset, the process fails to start. The UUID must come from
a prior out-of-band `POST /api/instances` call made by a human or another
script. There is no path inside the agent for that registration to happen.

**Desired behavior:**
The agent's required identity inputs are `project_name` and `hostname`
(the latter can default to the OS hostname). On startup:

1. If a previously-persisted instance id exists in the agent's local state
   directory, the agent uses it directly (idempotent restart â€” no extra
   API call, no duplicate rows in the control DB).
2. Otherwise, the agent calls the control plane's registration endpoint
   with its `project_name` and `hostname`, receives an instance id back,
   persists it to local state, and proceeds to subscribe.
3. If the control plane is unreachable on first start, the agent retries
   with bounded backoff before exiting non-zero. On subsequent starts
   (when the id is already persisted) the control plane being unreachable
   does not block startup â€” the agent still subscribes to NATS and the
   control plane learns about it through the heartbeat channel (see #003).

The `AXIS_AGENT_INSTANCE_ID` env var is removed from the required set. It
remains accepted as an *override* for advanced use, but is no longer the
primary path. The example `.env` and README quickstart are updated to
reflect the new flow (set `project_name` + `hostname`, start the agent,
done).

**Key interfaces:**

- Agent settings â€” `instance_id` becomes optional; `project_name` and
  `hostname` become required; `control_plane_url` (or equivalent) is added
  so the agent knows where to POST.
- Local state â€” the agent persists its assigned id to a single small file
  under a configurable state directory (default platform-appropriate, e.g.
  `~/.local/state/axis-agent/instance.json` on Linux). File format should
  include the id, the project name it registered under, and the timestamp.
- Control-plane registration endpoint â€” already exists
  (`POST /api/instances`). Should be safe to call again with the same
  hostname + project; today it creates a new row each time, which is
  acceptable because the agent only calls it when it has no persisted id.
  A future issue may add `(project_name, hostname)` uniqueness; out of
  scope here.
- Agent CLI / module entrypoint â€” accepts a flag (or env var) to reset
  the local state file, useful for "I want to re-register".

**Acceptance criteria:**

- [ ] An agent started with only `project_name`, `hostname`, and
      `control_plane_url` (no `instance_id`) registers itself successfully
      and begins receiving commands within seconds.
- [ ] Stopping and restarting that agent does not register a second
      instance â€” the local state file is honoured and the same UUID is
      reused.
- [ ] Deleting the local state file and restarting causes the agent to
      register again (producing a new row).
- [ ] If `AXIS_AGENT_INSTANCE_ID` is explicitly set, it overrides both the
      persisted state and the self-registration step (backward-compat).
- [ ] If the control plane is unreachable during first-time registration,
      the agent retries with bounded backoff (e.g. up to ~30 s total) and
      exits with a clear log message if still unreachable.
- [ ] README quickstart no longer requires the user to copy a UUID by
      hand â€” `uv run axis-agent` works directly after editing
      `project_name`/`hostname` in `.env`.
- [ ] Integration test: spin up control plane + an agent configured only
      with project/hostname, assert that a `POST /commands` to the
      auto-assigned instance id reaches the agent and gets a completed
      status back.

**Out of scope:**

- Authentication / token handling between agent and control plane â€” the
  registration call is unauthenticated for now. Auth lands in its own
  issue once the design is settled.
- `(project_name, hostname)` uniqueness on the control plane â€” see above.
- Migrating existing manually-registered instances. The dev DB will be
  wiped during this change; production has no users yet.
- Heartbeat / online status changes â€” covered by #003.
