# #002 — Commands published with no NATS subscriber stay pending forever

> *This was generated by AI during triage.*

**Category:** bug
**State:** ready-for-agent

## Reporter context

`POST /api/instances/{NEW_UUID}/commands {"type":"disable"}` against an
instance that has no live agent subscriber returns `202 Accepted` with
`status: "pending"` — and stays that way indefinitely. There is no
visible failure, no log, no eventual timeout. The same call against an
instance whose agent *is* connected completes within milliseconds.

This is the same root cause that surfaces #001 to the user, but it is
also its own bug: even after #001 lands, anyone whose worker has crashed,
been firewalled off, or is mid-restart will silently drop commands.

## Reproduction trace

- The HTTP route delegates to a dispatcher that does two things in order:
  persist a pending command row, then publish a `CommandMessage` on
  `commands.<instance_id>`.
- The NATS publish uses the **core** publish primitive — fire-and-forget,
  no acknowledgement, no queue, no replay. If zero subscribers exist on
  the subject at publish time, the broker drops the message.
- The pending command row has no expiry, no watchdog, no sweeper. It sits
  in `commands.status = 'pending'` until something explicitly updates it.
- The only updater today is the inbound status subscriber, which only
  fires when a real agent posts a `StatusMessage`. No agent → no update.
- Net effect: the row is now a permanent lie. Querying `/api/commands/{id}`
  still says `pending`. Querying `/api/instances/{id}` shows whatever
  status it had before. The maintainer has no way to tell the difference
  between "command in flight" and "command lost forever".

## Agent Brief

**Category:** bug
**Summary:** Pending commands must reach a terminal state within a bounded
time, regardless of whether an agent ever consumes the NATS message.

**Current behavior:**
Pending commands rely entirely on a live NATS subscriber and a cooperating
agent that publishes a status report. Either side absent → the command is
stuck at `pending` and the instance state remains stale.

**Desired behavior:**
Pending commands have a deadline. After a configurable timeout (default
on the order of 30–60 s — pick a sensible value, the actual number is not
load-bearing here), any command still in `pending` is moved to a `failed`
terminal state with a structured reason indicating that no acknowledgement
was received in time. The instance status is **not** flipped on timeout —
we don't know what actually happened on the worker, so the safest state
is "leave the instance status as it was and surface the failure on the
command".

The dispatch path should also do a best-effort up-front check: if there
are no current subscribers on the target subject when the command is
about to be published, the API should still accept the command (write it
as pending, schedule its timeout) but the response should include a
machine-readable hint that no agent is currently reachable, e.g. a
`delivery: "no_listeners"` or equivalent indicator. This gives the
operator immediate feedback in the common "I forgot to start the agent"
case without changing the contract for the queued / retried scenarios.

An alternative implementation path is to switch the commands subject to
NATS JetStream so the broker queues commands and replays them when an
agent connects. This trades a moving part (JetStream) for stronger
guarantees and is worth considering, but the timeout-and-fail path is
sufficient for the immediate bug.

**Key interfaces:**

- The dispatch result type — gains a delivery hint field (enum: e.g.
  `delivered_now`, `no_listeners`, `unknown`). The pending command row
  is written either way; the hint is informational.
- The command-status field — values currently include `pending`,
  `completed`, `failed`. Behaviour change: `failed` may now be set by a
  timeout sweeper as well as by an inbound status report.
- A new background component — call it whatever fits the layering, but
  conceptually it's a periodic task that finds rows where
  `status = 'pending'` and `issued_at < now - timeout`, marks them
  `failed`, sets `completed_at = now`, and records a stable reason
  string ("no acknowledgement received within timeout").
- NATS client — needs to expose subscriber-count information (NATS Core
  has a way to query this; check the client library) so the up-front
  hint can be populated without significantly slowing the publish path.

**Acceptance criteria:**

- [ ] Issuing a `disable` to an instance with no live subscriber returns
      a response that clearly indicates no listener was reachable at
      publish time.
- [ ] That same command transitions from `pending` to `failed` within the
      configured timeout window, *without* any agent ever connecting.
- [ ] The failure reason on a timed-out command is a stable string the
      UI can match on, not a free-form prose blob.
- [ ] If an agent connects after the timeout and *then* publishes a
      success status, the late status is logged as an anomaly but does
      not flip the command back to `completed` (terminal means terminal).
- [ ] Integration test: register an instance, do not start an agent,
      issue a command, advance time (or use a short timeout in test
      config), assert the command is `failed` with the documented reason.
- [ ] Integration test: existing happy-path tests still pass — commands
      that get a real status back within the window land as `completed`,
      not `failed`.
- [ ] The timeout is configurable via the control-plane settings, with
      a sensible default.

**Out of scope:**

- Migrating to NATS JetStream — listed as an alternative above; if the
  implementer chooses it, that is fine, but it is not required.
- Retrying failed commands automatically — operator-initiated retry only
  for now.
- Flipping instance status on timeout. We deliberately do *not* infer
  the worker state from a lost ack.
- Surfacing this in a UI — the UI lands in its own issue once the data
  is reliable.

## Triage notes

- This is the bug that makes #001's UX feel broken. With #001 alone the
  happy path works, but every misconfigured / dead worker still silently
  swallows commands. This issue keeps the system honest.
- Pairs naturally with #003: timeout-failed commands plus a heartbeat
  channel give the operator a true picture of "what is up vs what we
  think is up".
